# JOURNAL.md

## 我的起点

在开始之前，先交代背景：我有相对丰富的AI辅助编程经验。

过去一年里，我带领团队搭建了一套完整的 AI 辅助开发流程。不是那种「让 ChatGPT 写个函数」的用法，而是一条从需求输入 → 多轮需求澄清 → 技术选型 → 架构设计 → 编码 → 代码审查 → 质量专家跑单测 → 自修正->任务状态更新的自动流转的完整闭环。我们通过 `/sf`skill来唤醒整个流程——本质上和 superpowers 的思路一样。区别是我们并不是TDD驱动的，而是先实现，然后由质量专家去补充单元测试，然后执行，修复未通过的测试用例。

所以我对 CC 的能力边界比较清楚。幻觉、任务遗漏、长对话上下文压缩丢进度——都是老朋友了。我也很清楚一套方法论：**CC 开发的最佳实践，是编码之前先把所有约束定死。** 需求文档、需求澄清、技术架构、接口设计、编码规范——这些东西在写第一行代码之前就必须到位。否则 CC 写出来的东西会非常飘，你改代码的时间比写代码的时间还长。

带着这个认知，我开始做 RepoRadar。

整个项目花了一天多。前三个小时想方向，后面的执行交给 CC。

## Day-by-Day

### 前三小时：定方向、定规则

选题想了三个小时。要求说「做出一个你真正想要的、能跑起来的小东西」，我最终选了「Git 仓库分析工具」——因为它天然要和外部系统（JGit）交互、要处理 LLM 不可信输出、有清晰的管线结构、最终产出是一份看得见的 HTML 报告。简单说：能让 CC 各种能力都有发挥空间，同时各种毛病也藏不住。

方向定了之后，我没有急着写代码。按我知道的最佳实践，先把所有约束定出来：

1. **PRD（产品需求文档）**—— CC 帮我出初稿，我逐条审改，定义了 7 个分析阶段、四类异常规则、LLM 语义分析的输入输出
2. **需求澄清与设计确认**—— 把 PRD 里的模糊点全部掰清楚：LLM 不可用时怎么办？单 JAR 部署怎么处理前端？Git 克隆失败怎么重试？
3. **接口设计文档**—— 每个 DTO 的字段、SSE 事件格式、管线各阶段的输入输出，全部白纸黑字写下来
4. **实施计划**—— 16 个 Task，5 个 Phase，明确执行顺序和依赖关系
5. **CLAUDE.md**—— 9 大类编码规范，几百条规则。包括阿里巴巴命名规约、异常处理、日志级别、集合初始化容量、安全红线……
6. **三个 Skills**—— `jgit-expert.md`（JGit API 正确样板 + 幻觉黑名单）、`llm-integration.md`（LLM 输出三步清洗 + 降级策略）、`architecture-guard.md`（分层依赖 + 技术栈锁定）
7. **一个 Command**—— `review-standards`（129 行，覆盖 9 大类的自动代码扫描）

> 做完这些，一行业务代码还没写。但我知道，这些前置工作做好了，后面 CC 的输出质量会完全不一样。这是我们团队之前的经验反复验证过的：**给 CC 的约束越清晰、越前置，它写出来的代码越接近你的期望；反过来，如果你一边写一边定规则，你会花大量时间在「改代码」而不是「写代码」上。**

### 第一天下午到晚上：编码执行

约束全部就位，开始让 CC 写代码。

执行顺序是严格按照实施计划来的：Maven 脚手架 → DTO 层 → CLI 入口 → GitCloner → MetadataExtractor → ModuleDetector → AnomalyFilter → VagueScoringEngine → LLM 客户端 → LLM Map/Reduce → 管线编排器 → 报告生成器。

因为规则都定好了，大部分模块的编码过程比较顺。但开发了一段时间后，我发现一个问题：**CC 还是在出一些不符合规范的代码。**

CLAUDE.md 里写了「禁止魔法值」，它还是硬编码了 `"connection failed"` 字符串。写了「方法不超过 80 行」，个别方法还是超了。不是每次都犯，但时不时冒出来一个。

我意识到光靠 CLAUDE.md 的「自觉遵守」是不够的——规则写得再细，CC 也可能忽略。于是我加了一层强制约束：

- **Pre-commit hook**—— `git commit` 时自动跑 P3C + SpotBugs + find-sec-bugs 扫描。不过就不让提交。

这层加上之后，效果立竿见影。后面再遇到不规范代码，CC 自己就能处理：提交 → hook 扫出问题 → CC 自己改 → 重新提交通过。我一句话都不用多说。

> 这是一个很重要的认知升级：**CLAUDE.md 是建议，hook 是法律。** 我们团队之前的 `/sf` 流程再完善，也依赖人记得去触发。Hook 不需要任何人记得它，它就在那里，每次 commit 都会执行。

但即使有了 hook，还是有几个地方被扎了。

**JGit API 幻觉**

这是被打脸最狠的一次。

我在 `jgit-expert.md` 里已经写了「严禁 `Runtime.exec("git ...")`」，CC 确实没用 Runtime。但它编了别的：`DiffEntry.getAddedLines()`、`repository.getBranches()`、`commit.getDiff()`——这些方法在 JGit 里根本不存在。编译直接报红。

我翻 JGit 6.8 源码找到正确的 API，然后把「幻觉 API → 正确替代」的对照表写进了 skill 的「常见 API 幻觉黑名单」章节。之后所有涉及 JGit 的代码——GitCloner、MetadataExtractor、ModuleDetector、AnalyzeCommand 的 gitconfig 读取——全部一次编译通过。

> 这件事验证了我之前的判断：**CLAUDE.md 里的规则是「方向约束」，它告诉 CC 不要往哪走；但 skill 里的样板代码是「路径约束」，它直接告诉 CC 该怎么走。** 两者缺一不可。光说「不要用错误 API」没用，你得把正确 API 的用法直接喂给它。

**record vs @Data**

DTO 最初用了 Java record，P3C 扫描报了问题。改成 Lombok `@Data` 之后，测试也要跟着改。改完我在 CLAUDE.md 里加了一条：「DTO 使用 `@Data` + Lombok，不使用 record」。一个规范修复，一次单独提交。

**跨厂商 LLM 兼容**

先用 DeepSeek 测试，一切正常。切到智谱 GLM-4，立刻挂了——GLM 的 API 响应里多了一个 `reasoning_content` 字段，Jackson 遇到未知字段直接抛异常。加 `@JsonIgnoreProperties(ignoreUnknown = true)` 修复。

然后又出问题：后端 DTO 用 `@JsonProperty("risk_level")` 映射 JSON，Vue 前端期望 `riskLevel`，中间序列化来回转换出了错。改成 `@JsonAlias` + `@JsonProperty` 双向兼容才解决。

> 每个 LLM 厂商的 API 返回格式都有微妙差异。这种跨厂商兼容问题，CC 是不可能提前预判的——它不知道 GLM 会返回 `reasoning_content`，也不知道前端 Vue 的 camelCase 习惯。这种 bug 只能靠实际测试暴露。

第一天结束时，78 个测试全绿，端到端验证通过：43 条提交 → 完整分析 → HTML 报告生成。

### 第二天：打磨 + 生产化

功能完成，开始打磨。

**SSE 实时进度页面**——之前管线有 SSE 后端但没有前端页面。这部分意外地顺利，CC 对 Vue3 的 CDN 写法很熟练。

**热力图 bug**——报告里的活跃热力图 y 轴没数据。我人眼看到的，CC 没发现。原因是 x/y 坐标映射写反了。这种 bug 很典型：**AI 写的代码功能正确但语义反了。** 它不知道「横轴是星期、纵轴是小时」这个业务含义，它只知道传两个参数。

**代理自动检测**——用 JAR 包在生产环境跑的时候，GitHub clone 失败了。排查发现我本机 `~/.gitconfig` 里配了代理，原生 `git clone` 会自动走，但 JGit 不读 gitconfig 的代理配置。我让 CC 实现了三级优先的代理检测：`--proxy` 参数 > 环境变量 > gitconfig 自动读取。实现过程中又踩了一个 JGit 坑，CC 又幻觉了一个 API，我翻文档才找到正确写法。

**LLM Map 批量化**——分析一个有 41 条异常提交的仓库时，我发现 Map 阶段在逐条调 LLM API——41 次请求。代码完全正确，功能正常运行，测试全过。但 41 次 API 调用，每次 2-3 秒，光这一步就两分钟。

我让它改成批量的，10 条一批。改完 41 次调用变 5 次。

> 这件事再次确认了一个判断：**CC 能把你想做的事做得又快又好，但它不会替你想该做什么。** 它不知道 41 次 API 调用是浪费，它只知道「遍历列表，对每个元素调 API」。性能优化、架构级判断，是需要人来看的。

**上下文压缩丢进度**

长对话到后期，CC 开始「失忆」：我让它更新 docs 下的设计文档，它做完 README 就收工了，忘了还有 5 个文件要更新。不是因为懒，是因为上下文被压缩了，之前讨论「要更新哪些文档」的决策被物理性地删掉了。这种丢失和「忘了」不一样——忘了你提醒一下能想起来，但压缩之后那段上下文已经不在了，你只能重新说一遍。

> 这是我之前在团队项目里就遇到过的问题，但这次感受特别强烈。我们团队的 `/sf` 流程同样是长上下文驱动，但我们做了一个关键设计：**不依赖 AI 记住，依赖文件系统记住。**
> 
> 具体来说，我们的 `/sf` 有三层自动持久化：
> 
> - **状态管理器**（`sf-state-manager.py`）：自动备份、验证、损坏时自动从备份恢复，最多保留 10 份历史
> - **Orchestrator Checkpoint**：每个阶段完成后强制写入 `.sf/results/`、`.sf/modules/`、`.sf/pipeline.json`，支持断点续传
> - **Hook 自动上下文**（`sf_auto_context.py` + `sf_context_aware.py`）：SF agent 被调用时自动创建上下文标记，用户下次输入时自动检测并恢复项目状态
> 
> 而当前 CC + superpowers 的组合没有这个能力。CC 原生有 `TaskCreate/TaskList/TaskUpdate`，但那是内存中的任务列表，对话结束就没了。Superpowers 提供的是 skills 和 workflow 模式指导，不会自动持久化任务状态。在这个项目里，我只能手动创建 `docs/任务跟踪.md`，然后靠提示词提醒 CC 每完成一个任务就去更新——效果类似，但是靠人提醒的，不是自动的。

最终交付：50 个 Java 文件，4100 行主代码，13 个测试类，65 次原子提交。

## 认知变化：Day 1 vs Day 2

### 之前的我

我们团队用 `/sf` 前缀驱动完整开发闭环，我知道 CC 的边界在哪里。我的核心理念是：**把规则写得足够细、把流程设计得足够完整，CC 就能按你的期望产出。**

### 现在的我

规则写得够细是对的，这从项目结果能看出来——前置 3 小时的规则定义，换来后面 CC 高效执行，65 次提交全部遵循 Conventional Commits 规范。这个方法论没有问题。

但这个项目让我看到了一个之前没想透的点：**规则的执行机制比规则本身更重要。** 而且这三层机制不是我一开始就设计好的，是踩了坑一步步加上去的：

1. **第一层：CLAUDE.md（规则层）**—— 编码之前就写好了。效果：大部分规范 CC 会遵守，但不是每次都遵守。魔法值、超长方法这些老毛病还是会冒出来。

2. **第二层：Skill（知识层）**—— 开发过程中被 JGit 幻觉扎了之后加的。不是告诉 CC 「不要犯错」，而是把正确答案直接喂给它。效果：JGit 相关代码从此一次编译通过。

3. **第三层：Hook（强制层）**—— 发现 CLAUDE.md 的规则还是拦不住之后加的。`git commit` 时自动扫描，不过就不让提交。效果：CC 自己就能完成「提交 → 扫出问题 → 修 → 重提」的闭环，我一句话不用多说。

加上项目里本来就有的运行时校验（LLM 输出经 Jackson 反序列化 + 枚举白名单，系统不会崩），实际上是四层。

> 认知变化：我以前认为关键是「把规则写好」。现在我认为关键是「让规则不可绕过」。而且这不是一次性设计出来的，**是在实战中踩一个坑加一层防御，逐步叠加出来的。** CLAUDE.md 是建议，Skill 是知识，Hook 是法律，运行时校验是保险。四层叠加，CC 想犯个错都难。

## 两个「原来还能这样」

### 1. 失败记录本身就是最好的 Skill 素材

`jgit-expert.md` 的「API 幻觉黑名单」是这样诞生的：CC 编了 `DiffEntry.getAddedLines()`，编译报错，我翻源码找到正确写法，把对照表写进 skill。后面又编了 `repository.getBranches()`、`commit.getDiff()`，每一次翻车都变成黑名单里的一行。

之后所有涉及 JGit 的代码全部一次编译通过。一份 101 行的 skill，省了不知道多少次「编译→报错→查文档→改→再报错」的循环。

> 我以前认为 skill 是用来教 AI 怎么做对的。这次发现，skill 更大的价值是记录 AI 怎么做错了。**每一条幻觉黑名单都是一次真实的编译错误，比任何「最佳实践」都精准。** 因为它不是凭空想象的，是从实际报错里提炼出来的。

### 2. Hook 形成了自纠正闭环，我一句话没说

加了 pre-commit hook 之后，发生了一件有意思的事：

CC 写了一段代码，我让它提交。Hook 自动跑 P3C 扫描，扫出两个魔法值（`"connection failed"` 和 `2`），commit 被拦住。CC 看到扫描结果，自己改了——提取成常量 `CONNECTION_FAILED_KEYWORD` 和 `PROXY_URL_PARTS_COUNT`——重新提交，通过。

**整个过程我说了一句话：「提交」。** 我没告诉它哪里有问题，没指出该改什么。它看到扫描结果，自己修，自己过。

> 这不是我在用 AI 写代码，这是 AI 在我设定的规则框架里自举。闭环能跑通，是因为 hook 是系统级的强制约束——它是我在开发过程中发现 CLAUDE.md 拦不住之后才加的，事实证明这个决定非常值得。如果靠 CLAUDE.md 来要求「提交前自查」，CC 很可能「自查」完直接通过。但 hook 的扫描结果它没法忽略，因为 commit 真的会被拦住。

## 一个「这玩意还不行」

### 上下文压缩丢进度

这没什么好美化的。长对话到后期，CC 的上下文被压缩，之前讨论的设计决策直接消失。它不是「忘了」，是那段记忆被物理删除了。你再怎么提醒也没用，只能重新说一遍。

我们团队之前用 `/sf` 流程时也遇到过同样的问题，但我们的解法是**把持久化做进流程本身**：

- Orchestrator 文档里写死了：「不依赖对话上下文保存进度 → 进度写入项目内 `.sf/` 目录」。每个阶段完成后，agent 必须把 SF_RESULT 写入 `.sf/results/{module}_{stage}.json`，orchestrator 必须更新 `.sf/pipeline.json`。这是流程规则，不是建议。
- 状态管理器自动做备份和校验，文件损坏了自动从最近一份备份恢复。
- Hook 在 agent 调用时自动创建上下文标记，下次进入时自动恢复。

这样即使上下文被压缩到什么都不剩，你只要能找到 `.sf/` 目录，就能断点续传。

但当前 CC + superpowers 的组合没有这个机制。CLAUDE.md 和 skills 是静态的规则，不会随开发进度更新；memory 是手动的，需要你主动写入。CC 原生的 `TaskCreate/TaskList/TaskUpdate` 只存在于内存中，对话结束就没了。没有一个「每个阶段自动把当前进度和决策持久化到文件」的能力。

在这个项目里，我只能手动创建 `docs/任务跟踪.md`，然后靠提示词提醒 CC 每完成一个任务就去更新。效果类似，但是人肉驱动，不是系统驱动。

本质上这不是上下文窗口大小的问题——窗口再大也有上限。这是一个**状态管理缺失**的问题。AI 辅助开发工具需要像数据库一样，关键状态先落盘再执行。我们的 `/sf` 流程做了这件事，CC 目前还没做。后续可以考虑扩展 superpowers 的 skill，加入自动创建任务跟踪文档、自动更新进度、同时更新记忆的能力——这个方向值得深挖。

## 再给一周

**1. 更多异常规则。** 现在只有巨型提交、易挥发文件、跨域提交、模糊提交四条。还可以加：提交间隔异常（凌晨突击）、单文件高频修改（热点代码）、提交消息与实际变更不匹配。规则本身不难，难的是调阈值——需要拿真实仓库数据验证。

**2. 增量分析。** 现在每次都是全量分析。如果仓库有 5000 条提交，会很慢。应该支持增量模式，缓存历史分析结果。这需要改管线结构，目前的设计没有考虑增量。

**3. 自动状态持久化。** 这是这次项目暴露出的最大缺口。我们团队之前在 `/sf` 流程中实现了完整的自动持久化——状态管理器（备份/校验/恢复）、Orchestrator Checkpoint（每个阶段强制写文件）、Hook 自动上下文管理。CC + superpowers 目前没有这个能力。后续可以考虑扩展 superpowers 的 skill，加入自动创建任务跟踪文档、自动更新进度、同时更新记忆的能力。这是一个值得深挖的方向。

## 「换个角色」思考

假设我的同事是一个产品经理，完全没碰过 CC，想用它做本职工作。

我会给他配这些：

**Skill：`pr-review`**——他不懂代码，但需要评审开发交付是否符合需求。这个 skill 让 CC 读 PR diff，对比 PRD 文档，生成一份「需求覆盖度报告」：哪些用户故事实现了、哪些跳过了、哪些和 PRD 描述不一致。他不需要看懂代码，只需要看懂这份报告。

**Skill：`competitor-audit`**——输入竞品 URL，CC 自动抓取公开信息（官网、应用商店、更新日志），生成结构化竞品分析：核心功能对比、定价策略、用户评价摘要。比他手动一个个看高效得多。

**Hook：`prd-consistency`**——每次更新 PRD 文档时自动检查：用户故事是否有验收标准、功能描述是否有优先级标注、是否有自相矛盾的描述。产品经理写文档容易遗漏这些结构化元素，hook 帮他兜底。

为什么这么配？产品经理的核心痛点不是「不会写」，而是「容易漏」和「难以对比」。CC 的价值不是替他写，而是做他做不到的全面扫描——和我这个项目里的 hook 思路一模一样。
